IF ONLY I HAD READ THESE BEFORE THE INTERVIEWS

发表于 2018-04-28 | 分类于 博文 | 评论：

此处列举了一些亡羊补牢相关的内容，在面试中遇到过，但在事后才了解到的东西。

我始终认为，大学四年过去后沉淀下来的东西才是最终要的.

事务的隔离级别

uncommit read

按例子来。比如说财务给员工A发工资，员工A正常的工资是100块钱，财务输入错误，打过去120块钱。A看到以为涨工资了很开心，但是财务及时的回滚了事务（因为此时事务尚未commit），又重新打了100元。

结果： A卡里有100元，但是A以为有120元。

bad：脏读

解决：使用commit read

commit read

A拿着卡去消费（卡里有100元），打算花100元买一盒万智牌DOM系列的现开盒，结账时系统读取到卡里的余额为100元，此时A的妻子B取出了88元。等系统扣款时发现卡里的钱不够了，A感到很困惑。

结果：一个事务内两个相同的查询返回了不同的数据

解决：使用repeatable read

repeatable read

A拿着卡（100元）去消费（10元），在支付时事务开启，不允许update修改。
B查询A的消费记录，看到消费了10元，于是要打印记录单。此时A又消费了50元（多了一条INSERT），B打印出来的记录单显示一共消费了60元，于是很困惑。

结果：系统可以正常扣款。但产生了幻读问题。

解决：使用serializable

serializable

serializable是最高的事务隔离级别，在此级别下，事务串行化顺序执行，即一个接一个的执行，不能并发执行。这种级别效率低下。

大多数数据库默认的事务隔离级别是Read committed，比如Sql Server , Oracle。Mysql的默认隔离级别是Repeatable read。

百度百科：事务隔离级别

事务

比如说网上购物的一次交易，大概会包含以下几个数据库操作：

  1. 更新库存信息
  2. 保存用户付款信息
  3. 生成订单保存到数据库
  4. 更新用户购物数量等

这一系列的操作就是一个事务。

原子性

事务包含的操作要么全部成功，要么全部失败。就是失败了就得回滚。

一致性

如果说一个数据保存在多个服务器上，如果它变更，则所有的服务器都要变更。

两个人进行交易，这两个人的金钱总量保持不变，这就是一致性。

A = A - 100
B = B +100

这两行必须合并成一个事务，要么do all，要么do none of them

隔离性

就是指上面那四个隔离级别

持久性

事务一旦提交，对数据库中数据的修改就是永久的，即使系统出现故障也不会影响
